PARSING ANOMALIES & DATA QUIRKS
================================
Tracks anomalies discovered across all question packs.
Add new findings here each time a new batch is reviewed.


STRUCTURAL / FORMAT ISSUES
--------------------------

[Pack 2] Backslash-escaped quotes inside CSV-quoted fields
  - The raw file uses \" to escape quotes within a quoted field (e.g., "If they did a show
    called \"Survivor\""), rather than the standard CSV convention of doubling ("").
  - Python's csv.reader requires escapechar='\\' to handle this correctly.
  - Without it, the reader misparses the field boundaries, causing entire lines to be skipped
    with an "odd answer fields" warning.
  - Also confirmed present in: Pack 1, Pack 3. Treat as universal — expect it in every pack.

[Pack 2] Missing pipe separator between answers
  - Some answer group blobs are missing the | delimiter between two answer tokens.
  - Examples (Pack 2):
      Line 1676: ANSWER PHONE=!PHONE  — should be: ANSWER PHONE|=!PHONE
      Line 2122: !booksmart !BOOKMARK — should be: !BOOKMARK (!booksmart was stray data)
  - Also seen in: Pack 1 (lines 24, 130, 267, 974, 976, 1149, 1570).
  - Must be manually corrected in the raw file before parsing.
  - The script will warn and skip questions where this causes an unparseable structure.

[Pack 2] Stray / orphaned data in answer column
  - Line 2102 contained a bare "7,6" with no associated question content.
  - Must be manually removed from the raw file before parsing.
  - NOTE: The script emits a WARNING (but does not skip) when a bare integer is detected
    as an answer, since bare integers are otherwise valid exact-match answers (e.g. years,
    ages). Review these warnings to distinguish legitimate bare-integer answers from stray data.

[Pack 1] Embedded editorial "was N pts" notes in answer groups
  - Point values were sometimes revised, with the original value noted inline.
  - Two observed formats:
      !OUTFIT w 5,0        — "w N" note, point value corrected to 0
      !COMPUTER - WAS 2 PTS.,2  — "- WAS N PTS." note embedded in answer text
  - Affected lines include: 968, 1280, 1561, 1569, 1573, 1722, 1735, 1793, 1865, 2224.
  - Must be manually corrected before parsing. Point value of 0 will cause the group to
    be ignored by the script; confirm whether to correct or remove these groups.

[Pack 1] Missing round column (blank field)
  - 24 lines have a completely empty round field (e.g., `0,,NNNN,...`).
  - Affected lines: 48, 69, 99, 121, 146, 150, 201, 331, 405, 942, 1246, 1301, 1403,
    1468, 1598, 1603, 1725, 1732, 1742, 1787, 1801, 2089, 2130, 2137.
  - Also seen in: Pack 3 (lines 22, 25, 27), Pack 4 (many lines).
  - Script warns and skips these. Must assign a round number manually if they are to be kept.

[Pack 3] `|E|` tag in round column with no integer — incomplete round field
  - Line 1081 and the block at lines 2029–2046 (18 lines) have `|E|2` as their round column
    value but are missing an integer round number entirely.
  - Note: `|E|` alone IS parseable (the digit `2` at the end is the round number here);
    verify whether these lines truly lack a round number or simply use a non-standard format.
  - Script warns and skips if no integer can be extracted.

[Pack 3] Wholesale structural corruption on line 1081
  - The entire line appears to be legacy/corrupt data: no `!` modifiers anywhere, answer
    groups contain raw labels like `AGE`, `HEIGHT`, `WEIGHT`, `STATUS SINGLE MARRIED`.
  - This line will likely produce a skip or garbage output — recommend manual removal.

[Pack 1] Stray single-letter tokens after pipes
  - Lines 616 and 1988: `|S ` appears as a lone letter token (e.g., `!GROWL|S !GROWLING`).
    The `S` is likely a stray character, not a valid answer token.
  - Line 976: `LATEI` — the `I` is likely a corrupted `|` pipe character.
  - Must be manually corrected.

[Pack 4] Large blocks of blank round fields
  - Lines 212–422 (~211 lines) and 841–885 (~40 lines) have entirely empty round fields.
  - Also isolated blanks at lines 56, 110, 150, 157.
  - Largest blank-round occurrence seen across any pack so far. Must assign round numbers
    manually or remove these questions if the round cannot be determined.

[Pack 4] Non-standard round field with extra segment (line 1032)
  - Round field is `|E|O7|2` — an extra `O7` segment is present, likely a corruption of
    the standard `|E|2` format. Parser will still extract `2` as the round number.

[Pack 4] Fully broken answer structure (line 527)
  - The entire answer block is raw comma-separated integers with no group text, no `!`
    tokens, and no parseable point-value pairing (e.g., `13,34,12,23,14,10,...`).
  - This line cannot be parsed and must be manually removed or reconstructed.

[Pack 4] `ex!WORD` stray text prefix before modifier (line 101)
  - `ex!HUSBAND` — the text `ex` appears directly before `!HUSBAND` with no separator.
  - Likely intended as `~EX !HUSBAND` or `!EX-HUSBAND`. Must be manually corrected.

[Pack 4] Double-pipe creating empty token (line 807)
  - `| |` — two consecutive pipe separators with only a space between them, producing an
    empty/whitespace-only token that the parser will treat as a no-op but is worth cleaning.


PREFIX FORMAT VARIATIONS
------------------------

[Pack 2] Standard format: first field is `5`
  - e.g., `5,|2,<ID>,<question>,...`

[Pack 4, Pack 6] .bine format: first field is `254`, round uses `|E|N` tag
  - e.g., `254,|E|2,<ID>,<question>,...`
  - The `E` tag in the round column should be ignored like W, M, T tags.

[Pack 5] Mixed prefix: first field is `0` for standard questions, `200` for holiday questions
  - e.g., `0,|2,<ID>,<question>,...` and `200,|1,<ID>,<question>,...`
  - The script ignores the first field entirely, so this should not affect parsing.
  - The `200` prefix block contains ~335 lines of holiday-specific questions at the end of
    the file. Be aware when assigning question packs — these may need a sub-pack distinction.


ROUND COLUMN TAGS
-----------------

[Pack 2] Letter tags appear alongside round numbers and must be ignored.
  - All observed formats across all packs: |2, |W|2, |M|3, |T|2, |2|T, |E|2, |W|3, |T|3
  - Rule: extract only the integer; discard any letter tags (W, M, T, E, etc.).

[Pack 3] `|E|` tag observed in round column
  - Line 1081: `|E|2` — the `E` tag is consistent with the .bine format prefix; treat same
    as other letter tags (ignore, extract the integer only).


CARET (^) OPTIONAL-WORD SYNTAX
-------------------------------

[Pack 1, Pack 4, Pack 5, Pack 6] The `^` caret is used as an optional-word marker mid-token.
  - Indicates that the word(s) following `^` are optional components of the answer phrase.
  - Examples:
      LIBERTY ^BELL              — "LIBERTY BELL" or just "LIBERTY" scores
      PAM ^LEE ANDERSON          — "PAM ANDERSON" scores even without middle name
      !MARK ^HIS ^HER TERRITORY  — any of "HIS"/"HER" may or may not appear
      ^GO ^GONE BANANAS          — "BANANAS" is the required word; GO/GONE optional
  - Present in Pack 1 (~40+ lines), Pack 4 (5 lines), Pack 5 (extensively, cleanly),
    Pack 6 (4 lines, in questions imported from other packs).
  - NOT present in Pack 2 or Pack 3.
  - The `^` ALSO appears inside corrupted/garbage trailing data in packs 1, 3, 4, and 6 —
    distinguish from valid use by context (garbage data is binary-looking character soup).
  - Will be resolved via manual cleanup. Answer matching pipeline implementation
	makes this syntax obsolete.


[Pack 5, Line 1565] Special case: `E=MC^2`
  - The `^` here is a superscript marker (mathematical notation), NOT an optional-word marker.
  - Should be preserved as literal text; treat the entire token as a single answer string.

[Pack 1, Pack 4, Pack 5] Asterisk `*` used alongside `^`
  - Observed: `!NOT OUT ^SIDE * DAY`, `!DOESN'T ^GO ^OUT ^SIDE * DAY`
  - Appears to function as a wildcard or "any word" placeholder in combination with `^`.
  - Extremely rare (2 lines in Pack 1, same lines repeated in Pack 4 and Pack 5).
  - Should be removed manually.


ANSWER MODIFIER EDGE CASES
---------------------------

[Pack 2] Double and combined modifier prefixes
  - !! (double exclamation)    -> treat as fuzzy (matchType 1)
  - !# and #!                  -> treat as fuzzy (matchType 1)
  - +!                         -> treat as fuzzy (matchType 1)
  - =! and !=                  -> both mean exact (matchType 2)
  - Bare integer answer text (no modifier at all) -> exact (matchType 2); triggers WARNING

[Pack 1] Additional modifier edge cases
  - !=! (triple modifier)      -> Should be removed manually.
  - !^ (bang-caret)            -> Should be removed manually.
  - $ (dollar sign modifier)   -> Should be removed manually.
  - # (standalone hash)        -> treat as fuzzy (matchType 1)
  - = (standalone equals, no !) -> treat as exact (matchType 2)
  - ! followed by space (e.g., `! CD PLAYER`) -> strip modifier and space; .strip() handles it

[Pack 3] `s!` stray prefix (line 27)
  - `s!SORE THROAT` — lowercase `s` prepended before `!`. Likely a data entry typo.
  - Manual correction: remove the stray `s`.

[Pack 3] `~!` combined forbidden-word + modifier (line 1169)
  - `~!DRIVING FAST/ RECKLESS` — tilde and bang combined as a prefix.
  - Unclear intent; treat `~` as forbidden-word marker and `!` as fuzzy modifier.

[Pack 2] Typo in modifier/answer
  - Line 1539: =crereal — likely a typo for =!CEREAL, but no auto-correction applied.
  - Parsed as an exact match for the literal text "crereal" per the warn-and-continue policy.
  - Watch for similar typos when reviewing future data.


ANSWER TEXT DATA ERRORS (trailing garbage)
------------------------------------------
NOTE: The script does not auto-strip these. They must be corrected manually in the raw
file before parsing. The script will warn and skip any question where trailing garbage causes
a structural parse failure.

[Pack 2] Known trailing characters observed on answer text:
  - Trailing =   e.g., TELEVISION=|   (line 1421)
  - Trailing !   e.g., UPGRADE!|      (line 1410)
  - Trailing '   e.g., CORN'

  - Must be manually removed from each affected line before parsing.


FORBIDDEN WORD FORMAT
---------------------

[Pack 2] Space between tilde and word
  - Standard format is ~word (no space), but ~ word (with space) also appears.
  - Both forms must be treated identically as forbidden word declarations.

[Pack 2] Multiple tilde tokens before one answer
  - Multiple consecutive ~word tokens can precede a single answer token; all are
    forbidden words for that answer.
  - Example: ~ship ~boat OCEAN  ->  answerText="OCEAN", forbiddenWords=["ship","boat"]

[Pack 1] Forbidden word in non-initial position
  - Line 1892: `!BOOTS ~BOOKS` — the `~BOOKS` exclusion appears after an answer token,
    not before it. Verify parser handles this (forbidden word should apply to the preceding
    answer token, not the next one).

[Pack 3] `!~` inverted sequence (line 867)
  - `!~shirt !SKIRT/ DRESS` — tilde follows bang instead of preceding it.
  - Intended as: `~shirt !SKIRT/ DRESS` (shirt is the forbidden word for SKIRT).
  - Must be manually corrected to move `~shirt` before the `!` of the next token.

[Pack 3] Forbidden word folded into answer text (line 960)
  - `=!big HAIR` — the lowercase `big` appears to be a forgotten `~big` forbidden word
    that was written as part of the answer text instead.
  - Must be manually corrected to: `~big =!HAIR`.
